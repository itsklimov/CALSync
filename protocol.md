# CALSync Protocol

CALSync agents exchange busy-time metadata through a private server. The protocol is platform-neutral: macOS is one client implementation, and other clients can interoperate without changing the server contract.

## Endpoints

### `GET /health`

Unauthenticated health probe.

Response `200`:

```json
{ "ok": true }
```

### `POST /v1/sync`

Authenticated sync endpoint. Requests must include:

```http
Authorization: Bearer <agent_token>
Content-Type: application/json
```

The bearer token is the group's sync code (`shared_secret` in client configs): a high-entropy string of at least 16 characters, generated by the first client in the group. The server derives the group identity from it:

```text
group_id = SHA-256(token)
```

Agents sending the same token form one isolated group: they see each other's plans and nothing from other groups. Unseen `agent_id`s are auto-registered into the token's group on their first valid sync. The server stores only the group hash, never the token; tokens shorter than 16 characters are rejected with `401` so trivially guessable groups cannot exist.

## Setup Code

Setup codes are base64-encoded UTF-8 JSON:

```json
{
  "server_url": "https://calsync.example.com",
  "shared_secret": "generated-64-char-hex-sync-code"
}
```

The setup code is only local pairing material for adding a Mac to an existing group; the embedded `shared_secret` is the group's sync code.

## Sync Request

```json
{
  "agent_id": "personal",
  "display_code": "PR",
  "mode": "delta",
  "window_days": 30,
  "upserts": [
    {
      "busy_id": "hash",
      "source_calendar_hash": "hash",
      "source_event_hash": "hash",
      "occurrence_hash": "hash",
      "start": "2026-05-21T17:00:00.000Z",
      "end": "2026-05-21T17:30:00.000Z",
      "all_day": false,
      "updated_at": "2026-05-20T17:00:00.000Z"
    }
  ],
  "deletes": ["busy_id"]
}
```

Fields:

- `agent_id`: stable agent identifier, non-empty string, max 256 characters.
- `display_code`: user-visible label placed on mirror events, non-empty after trimming.
- `mode`: `full` or `delta`.
- `window_days`: positive integer, max 366.
- `upserts`: array of busy blocks, max 10,000.
- `deletes`: array of `busy_id` strings, max 10,000.
- Timestamps: strict UTC ISO-8601 accepted by both JavaScript `Date.parse` and SQLite `julianday`: `YYYY-MM-DDTHH:mm:ssZ` or `YYYY-MM-DDTHH:mm:ss.SSSZ`.
- Busy-block string fields are non-empty strings, max 256 characters.
- `end` must be later than `start`.
- `all_day` must be boolean.

Upload-only privacy fields are accepted only in requests and are not returned to other agents: `source_calendar_hash`, `source_event_hash`, `occurrence_hash`, `all_day`, and `updated_at`.

## Sync Response

```json
{
  "blocks": [
    {
      "busy_id": "hash",
      "display_code": "WX",
      "start": "2026-05-21T17:00:00.000Z",
      "end": "2026-05-21T17:30:00.000Z"
    }
  ],
  "state_hash": "lowercase-sha256-hex"
}
```

`blocks` contains active blocks from other non-stale agents inside the requested managed window. It never includes the caller's own blocks.

## Full and Delta Semantics

`full` mode replaces the caller's active state inside the managed window `[received_at, received_at + window_days)`. Incoming `upserts` are active after the run. Previously active caller blocks that overlap the managed window and are not present in the full request are marked inactive. Previously stored blocks outside that window are left alone.

`delta` mode applies incremental changes only. Incoming `upserts` insert or replace matching `(agent_id, busy_id)` rows and mark them active. `deletes` mark only those caller `busy_id` rows inactive. Unmentioned active rows remain active.

The server updates agent liveness on successful sync. Plans omit blocks from agents whose `last_seen_at` is older than the configured TTL.

## State Hash

`state_hash` describes the server's active state for the caller inside the requested managed window. It is used by agents to detect drift and choose `full` when local outgoing state disagrees.

Algorithm:

1. Select active rows for the caller where `end > now` and `start < now + window_days`.
2. Sort by `busy_id` ascending.
3. For each row, render:

   ```text
   <busy_id>:<start>:<end>:<all_day_as_0_or_1>
   ```

4. Join rows with `|`.
5. Return SHA-256 of the UTF-8 payload as lowercase hex.

The empty state hash is SHA-256 of the empty string.

## Privacy Model

Agents never upload event titles, locations, notes, attendees, calendar names, raw event identifiers, or raw calendar identifiers. Agents upload only:

- Stable hashed IDs scoped by the agent privacy salt.
- Start and end timestamps.
- `all_day`, used only for sender state hashing.
- A display label chosen by the user.

The server stores hashed identifiers, active window state, timestamps, display labels, group hashes, and liveness metadata. It has enough information to coordinate busy intervals but not to reconstruct local calendar content.

Responses intentionally return only `busy_id`, `display_code`, `start`, and `end`.

## Mirror Markers

Mirror events are slot-based. A slot is an exact normalized `(start, end)` interval. Its marker hash is:

```text
SHA-256("<start>|<end>")
```

The primary marker is the event URL:

```text
calsync://slot/<hash>
```

Some calendar backends drop custom event URLs. Clients must also write and recognize a notes fallback on the first line:

```text
calsync-slot:<hash>
```

Legacy pre-slot agents used:

```text
calsync://busy/<id>
```

Clients must treat `calsync://busy/` markers as deletion-only legacy markers. They are recognized so they can be removed, but new clients must not create them.
